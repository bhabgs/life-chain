# 多平台适配说明

> 本项目的文件保护机制支持 GitHub、GitLab、Gitee 等主流 Git 平台

---

## 📊 兼容性矩阵

| 保护机制 | GitHub | GitLab | Gitee | 其他 Git | 说明 |
|---------|--------|--------|-------|----------|------|
| **Git Hooks** | ✅ | ✅ | ✅ | ✅ | Git 原生功能，全平台支持 |
| **CODEOWNERS** | ✅ | ✅ | ✅ | ❌ | 主流平台支持 |
| **CI 自动检查** | ✅ | ✅ | ✅ | ⚠️ | 需要配置对应平台的 CI |

---

## 🎯 通用保护层（所有平台）

### 1. Git Hooks - 本地保护（100%兼容）

Git Hooks 是 Git 原生功能，**完全兼容所有平台**。

#### 文件位置
```
.git/hooks/pre-commit
```

#### 安装方式
```bash
# 通用安装脚本
bash scripts/install-hooks.sh
```

#### 工作原理
- 本地提交前自动检查
- 与远程仓库平台无关
- 完全在本地执行

#### 特点
- ✅ 所有 Git 仓库都支持
- ✅ 无需远程配置
- ✅ 即时生效
- ⚠️ 可以被 `--no-verify` 绕过
- ⚠️ 需要每个开发者手动安装

---

## 🌐 平台特定保护层

### 2. CODEOWNERS - 代码所有者

不同平台对 CODEOWNERS 的支持：

#### GitHub
**文件位置**: `.github/CODEOWNERS` 或 `CODEOWNERS`（根目录）

**语法**:
```
# 格式：路径  @用户名
CLAUDE.md  @project-owner @tech-lead
docs/*.md  @tech-lead
```

**文档**: https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-code-owners

#### GitLab
**文件位置**: `.gitlab/CODEOWNERS` 或 `CODEOWNERS`（根目录）

**语法**:
```
# 格式：路径  @用户名
CLAUDE.md  @project-owner @tech-lead
docs/*.md  @tech-lead
```

**配置要求**:
- GitLab Premium 或更高版本
- 需要在项目设置中启用

**文档**: https://docs.gitlab.com/ee/user/project/codeowners/

#### Gitee（码云）
**文件位置**: `CODEOWNERS`（仅支持根目录）

**语法**:
```
# 格式：路径  @用户名
CLAUDE.md  @project-owner @tech-lead
docs/*.md  @tech-lead
```

**配置要求**:
- Gitee 企业版支持
- 需要在项目设置中启用"保护分支"

**文档**: https://gitee.com/help/articles/4369

#### 其他平台
- **Bitbucket**: 使用 `CODEOWNERS` 在根目录（Cloud 版本支持）
- **Azure DevOps**: 不支持 CODEOWNERS，需要使用分支策略
- **自建 Git**: 通常不支持

### 3. CI 自动检查

不同平台的 CI 配置：

#### GitHub Actions
**文件位置**: `.github/workflows/check-protected-files.yml`

**触发方式**:
```yaml
on:
  pull_request:
    branches:
      - main
      - develop
```

#### GitLab CI
**文件位置**: `.gitlab-ci.yml`

**配置示例**:
```yaml
check-protected-files:
  stage: test
  script:
    - bash scripts/check-protected-files.sh
  only:
    - merge_requests
```

#### Gitee Go
**文件位置**: `.gitee/workflows/check-protected-files.yml`

**配置示例**:
```yaml
name: 检查受保护文件
on:
  pull_request:
    branches:
      - master
      - develop

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: bash scripts/check-protected-files.sh
```

#### Jenkins / 其他 CI
需要配置 Jenkinsfile 或对应的 CI 配置文件，调用检查脚本。

## 🔧 通用化改造方案

### 方案 1：仅使用 Git Hooks（推荐，100%兼容）

**优点**:
- ✅ 所有平台都支持
- ✅ 无需平台特定配置
- ✅ 简单易用

**缺点**:
- ⚠️ 可以被绕过
- ⚠️ 需要每人安装

**适用场景**:
- 小团队
- 内部项目
- 无法使用平台高级功能

### 方案 2：Git Hooks + CODEOWNERS（推荐，主流平台）

**优点**:
- ✅ 本地 + 远程双重保护
- ✅ GitHub/GitLab/Gitee 都支持
- ✅ 强制审查

**缺点**:
- ⚠️ 需要平台高级功能
- ⚠️ 配置略复杂

**适用场景**:
- 中大型团队
- 商业项目
- 使用主流平台

### 方案 3：完整保护（最强，需要 GitHub/GitLab）

**包含**:
- Git Hooks
- CODEOWNERS
- CI 自动检查

**优点**:
- ✅ 三层保护
- ✅ 最严格
- ✅ 完全自动化

**缺点**:
- ⚠️ 仅限 GitHub/GitLab
- ⚠️ 配置复杂

**适用场景**:
- 大型项目
- 开源项目
- 严格质量要求

## 📝 不同平台的配置步骤

### GitHub 配置

```bash
# 1. 保持现有结构
.github/
  ├── CODEOWNERS
  ├── hooks/
  │   └── pre-commit
  └── workflows/
      └── check-protected-files.yml

# 2. 安装 Hooks
bash scripts/install-hooks.sh

# 3. 在 GitHub 上启用
# Settings → Branches → 添加分支保护规则
# 勾选：Require a pull request before merging
# 勾选：Require review from Code Owners
```

### GitLab 配置

```bash
# 1. 调整目录结构
.gitlab/
  ├── CODEOWNERS
  └── hooks/
      └── pre-commit

.gitlab-ci.yml

# 2. 安装 Hooks
bash scripts/install-hooks.sh

# 3. 在 GitLab 上启用
# Settings → Repository → Protected branches
# Settings → Merge requests → Code Owners
```

### Gitee 配置

```bash
# 1. 调整目录结构（Gitee 要求 CODEOWNERS 在根目录）
CODEOWNERS           # 必须在根目录
.gitee/
  ├── hooks/
  │   └── pre-commit
  └── workflows/
      └── check-protected-files.yml

# 2. 安装 Hooks
bash scripts/install-hooks.sh

# 3. 在 Gitee 上启用
# 设置 → 保护分支 → 添加保护分支
# 设置 → 代码审查 → 启用 CODEOWNERS
```

### 自建 Git 服务器 / 其他平台

```bash
# 只使用 Git Hooks
hooks/
  └── pre-commit

# 安装 Hooks
bash scripts/install-hooks.sh

# 可选：使用服务器端 Hooks
# 在 Git 服务器上配置 pre-receive 或 update hooks
```

## 🔄 迁移指南

### 从 GitHub 迁移到 GitLab

```bash
# 1. 移动 CODEOWNERS
mv .github/CODEOWNERS .gitlab/CODEOWNERS

# 2. 转换 CI 配置
# 将 .github/workflows/*.yml 转换为 .gitlab-ci.yml

# 3. 重新安装 Hooks
bash scripts/install-hooks.sh
```

### 从 GitHub 迁移到 Gitee

```bash
# 1. 移动 CODEOWNERS 到根目录
mv .github/CODEOWNERS ./CODEOWNERS

# 2. 移动 workflows
mkdir -p .gitee/workflows
cp .github/workflows/*.yml .gitee/workflows/

# 3. 调整配置文件中的分支名（master vs main）

# 4. 重新安装 Hooks
bash scripts/install-hooks.sh
```

## ⚠️ 注意事项

### 1. CODEOWNERS 文件位置

不同平台要求不同：
- GitHub: `.github/CODEOWNERS` 或根目录
- GitLab: `.gitlab/CODEOWNERS` 或根目录
- Gitee: **仅支持根目录**

**建议**: 使用根目录的 `CODEOWNERS`，兼容性最好。

### 2. 分支名称差异

- GitHub 默认: `main`
- Gitee 默认: `master`
- GitLab 默认: `main` 或 `master`

**建议**: 在配置文件中同时支持两者。

### 3. CI 语法差异

不同平台的 CI 配置语法不同，需要分别编写。

**建议**: 使用通用的 bash 脚本，在不同 CI 中调用。
