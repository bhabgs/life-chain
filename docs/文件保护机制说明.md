# 文件保护机制说明

> 本文档说明生命链项目的文件保护机制

---

## 📋 概述

为了确保项目的稳定性和一致性，我们对关键文件实施了多层保护机制。

### 保护目标

- ✅ 防止意外修改关键配置
- ✅ 确保重要变更经过审查
- ✅ 保持团队知晓重要修改
- ✅ 维护文档和代码的一致性

### 保护层级

```
┌─────────────────────────────────────────┐
│     第一层：本地 Git Hooks              │
│     (提交前自动检查)                    │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│     第二层：GitHub Actions CI           │
│     (PR 时自动检查)                     │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│     第三层：CODEOWNERS                  │
│     (强制代码审查)                      │
└─────────────────────────────────────────┘
```

## 🔧 保护机制详解

### 1. Git Hooks（本地保护）

#### 工作原理

```bash
git commit
    ↓
Pre-commit Hook 执行
    ↓
检查暂存区文件
    ↓
是否包含受保护文件？
    ↓
是 → 显示警告 → 要求确认
    ↓
否 → 直接通过
```

#### 安装方式

```bash
# 首次安装
bash scripts/install-hooks.sh

# 检查是否安装
ls -la .git/hooks/pre-commit
```

#### 警告示例

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️  警告：你正在修改受保护的文件！
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下文件受到保护，需要特定权限人员审批：

  ✗ CLAUDE.md
  ✗ docs/技术架构设计.md

这些文件的修改需要：
  1. 项目负责人或技术负责人审批（CODEOWNERS）
  2. 详细的修改说明（在 PR 中）
  3. 团队成员知晓

是否确认提交这些修改？[y/N]:
```

#### 绕过方式（仅限紧急情况）

```bash
git commit --no-verify -m "紧急修复"
```

⚠️ **警告**：不要频繁使用 `--no-verify`，这会绕过所有检查。

### 2. GitHub Actions（CI 保护）

#### 工作流程

```yaml
PR 创建/更新
    ↓
触发 Workflow
    ↓
检测变更文件
    ↓
包含受保护文件？
    ↓
是 → 执行检查流程
    ├─ 添加警告评论
    ├─ 检查 PR 描述
    ├─ 添加标签
    └─ 描述不完整 → 阻止合并
    ↓
否 → 通过检查
```

#### 检查项目

1. **文件检测**
   - 使用 `tj-actions/changed-files`
   - 检测受保护文件的变更

2. **PR 评论**
   - 列出修改的文件
   - 提供检查清单
   - 链接相关文档

3. **描述检查**
   - 长度 > 100 字符
   - 包含关键词：`修改原因`、`为什么`、`变更说明`

4. **标签管理**
   - 自动添加：`需要特别审查`
   - 自动添加：`受保护文件修改`

#### 示例输出

```markdown
## ⚠️ 受保护文件修改警告

这个 PR 修改了受保护的文件，需要特别审查。

### 📋 修改的受保护文件：
- `CLAUDE.md`
- `docs/技术架构设计.md`

### ✅ 检查清单：
- [ ] 已与相关负责人沟通
- [ ] 在 PR 描述中详细说明了修改原因
- [ ] 已添加必要的审查人员
- [ ] 修改已在团队中知晓
- [ ] 相关文档已同步更新

### 📚 相关文档：
- [CODEOWNERS](.github/CODEOWNERS)
- [CLAUDE.md](CLAUDE.md)
```

### 3. CODEOWNERS（审查保护）

#### 工作原理

GitHub 会根据 CODEOWNERS 文件自动：
1. 添加必需的审查者
2. 要求所有指定审查者批准
3. 阻止未经审查的合并

#### 配置格式

```
# 文件路径模式  审查者1  审查者2

CLAUDE.md                    @project-owner @tech-lead
docs/*.md                    @tech-lead
**/prisma/schema.prisma      @backend-lead @dba
```

#### 审查流程

```
PR 创建
    ↓
CODEOWNERS 匹配
    ↓
自动添加审查者
    ↓
等待审查者批准
    ├─ 批准 → 允许合并
    ├─ 请求修改 → 修改后重新审查
    └─ 拒绝 → 关闭 PR
```

## 📝 修改受保护文件的标准流程

### 流程图

```
1. 事前沟通
   ↓
2. 创建分支
   ↓
3. 修改文件
   ↓
4. 本地提交（Hook 检查）
   ↓
5. 推送到远程
   ↓
6. 创建 PR
   ↓
7. CI 自动检查
   ↓
8. 审查者审查
   ↓
9. 所有审查者批准
   ↓
10. 合并到主分支
```

### 详细步骤

#### 步骤 1：事前沟通

- [ ] 与相关负责人讨论修改必要性
- [ ] 在团队会议中说明修改计划
- [ ] 获得初步同意
- [ ] 确定修改范围

#### 步骤 2：创建分支

```bash
git checkout -b feature/update-protected-file
```

#### 步骤 3：修改文件

- [ ] 进行必要的修改
- [ ] 确保修改经过充分测试
- [ ] 更新相关文档

#### 步骤 4：本地提交

```bash
git add docs/技术架构设计.md
git commit -m "[PROTECTED] docs: 更新技术架构设计文档

修改原因：
- 技术栈从 XXX 升级到 YYY
- 架构从单体调整为微服务

影响范围：
- 后端开发团队需要了解新架构
- 部署流程需要更新

审查者：@tech-lead @project-owner"
```

Hook 会显示警告，确认后继续。

#### 步骤 5：推送到远程

```bash
git push origin feature/update-protected-file
```

#### 步骤 6：创建 PR

使用以下模板：

```markdown
## 📋 修改的受保护文件

- [x] docs/技术架构设计.md

## 📝 修改原因

### 背景
XXX

### 目标
XXX

### 修改内容
1. XXX
2. XXX

## 📊 影响范围

- 影响的模块：XXX
- 影响的团队成员：XXX
- 需要的后续行动：XXX

## ✅ 检查清单

- [x] 已与 @tech-lead 沟通
- [x] 修改已在团队中知晓
- [x] 相关文档已同步更新
- [x] 已添加必要的审查人员

## 👥 审查者

@tech-lead @project-owner
```

#### 步骤 7-8：CI 检查和审查

- CI 自动添加评论和标签
- 审查者收到通知
- 审查者查看修改并批准/请求修改

#### 步骤 9-10：合并

所有审查者批准后，即可合并。

## 🚨 紧急情况处理

### 场景：生产环境紧急修复

如果遇到生产环境问题需要紧急修改受保护文件：

1. **立即行动**
   ```bash
   # 创建紧急修复分支
   git checkout -b hotfix/urgent-fix

   # 修改文件
   vim CLAUDE.md

   # 提交（可以使用 --no-verify）
   git commit --no-verify -m "[URGENT] fix: 紧急修复生产环境配置

   问题：XXX
   修复：XXX
   影响：XXX"

   # 推送
   git push origin hotfix/urgent-fix
   ```

2. **立即通知**
   - 在团队群中说明情况
   - @ 项目负责人和技术负责人
   - 标注为紧急

3. **创建 PR**
   - 标题添加 `[URGENT]` 前缀
   - 详细说明紧急情况
   - 请求快速审查

4. **快速审查**
   - 项目负责人快速审查
   - 必要时可以单人批准合并

5. **事后处理**
   - 在下次团队会议中说明
   - 补充完整的文档
   - 总结经验教训

## 🔍 监控与审计

### 定期检查

每月进行一次受保护文件修改审计：

1. **统计修改频率**
   ```bash
   git log --all --since="1 month ago" --name-only --pretty=format: | \
   grep -E "(CLAUDE.md|docs/.*\.md|package.json)" | \
   sort | uniq -c | sort -rn
   ```

2. **检查绕过行为**
   ```bash
   git log --all --grep="--no-verify" --since="1 month ago"
   ```

3. **审查修改原因**
   - 每次修改是否有充分理由
   - 是否经过适当审查
   - 是否通知团队

### 审计报告

每季度生成一次审计报告，包含：
- 受保护文件修改次数
- 修改原因统计
- 绕过检查次数
- 改进建议

## ❓ 常见问题

### Q1: 我只是修改了一个错别字，也需要走完整流程吗？

**A**: 是的。即使是小修改，也需要确保：
- 修改是正确的
- 团队知晓修改
- 修改有记录

不过对于明显的错别字修改，审查会很快通过。

### Q2: 审查者不在线，怎么办？

**A**:
1. 联系项目负责人临时指定其他审查者
2. 如果非常紧急，可以走紧急流程
3. 正常情况下，耐心等待（通常 24 小时内会有反馈）

### Q3: 我可以修改 CODEOWNERS 吗？

**A**: 可以，但需要：
1. 项目负责人批准
2. 充分的理由
3. 在团队中讨论

### Q4: Hook 检查失败，如何修复？

**A**:
1. 检查 hook 是否正确安装
2. 重新安装：`bash scripts/install-hooks.sh`
3. 确认文件权限：`chmod +x .git/hooks/pre-commit`

### Q5: 可以禁用这些检查吗？

**A**: 不建议。如果确实需要：
1. 必须有项目负责人批准
2. 记录在案
3. 定期审查是否需要恢复
